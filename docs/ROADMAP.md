## The product vision / roadmap (the real build)
The north star: **point it at any website; it onboards itself, then tests like real users.**
1. **Explorer role (distinct from testing personas)** — probes a site for affordances (signup? payments? logs? API? MCP? could the human grant read-only DB access?), writes up its understanding, and generates a **per-site human questionnaire** (unbounded; each site differs) surfaced in the UI. Re-runnable.
   - **Affordance discovery, NOT intrusion**: the agent *proposes* capabilities, the human *grants* them. It never attempts unauthorized access. (Ethics/legality for "any site" + cleaner model.)
   - The questionnaire is the product's hinge: it unifies consent/authorization + configuration (creds/scope) + knowledge-elicitation in one adaptive artifact.
2. **Secrets vault** — questionnaire answers flagged sensitive (DB passwords, test logins) go to an encrypted per-target vault; the Site Model stores only `credential_ref` *pointers*. Build this boundary before storing any secret. Store the dynamic questionnaire as a `site_questions` collection; give each target a lifecycle (`registered → exploring → awaiting-answers → configured → testing → re-explore`).
   - ✅ **Vault store layer built** (`qa_store/vault.py` + the `site_secrets` collection): `put_secret`/`get_secret`/`secret_exists`/`list_secret_refs` (metadata-only, never values)/`delete_secret`, values encrypted via `qa_store.crypto` (Fernet, `QA_CREDENTIAL_KEY`). Opaque `vault://<tenant>/<target>/<ref>` pointers; `get_secret` is the only path that returns a raw value. This is the boundary — built before any consumer stores a secret.
   - ✅ **Questionnaire + lifecycle store layer built** (`qa_store/site_questions.py` + the `site_questions` collection; lifecycle on `site_targets`). Questions are rows keyed by `(tenant, target, question_id)` with `kind` (`free_text`/`secret`/`choice`/`boolean`/`url`/`number`), `category`, `rationale`, `required`, `order`, `status` (`open`/`answered`/`skipped`). `answer_site_question` routes a **secret** answer into the vault and stores only the `credential_ref` — the raw value never touches the question row; non-secret answers store inline. Re-exploring re-`upsert`s questions (refreshes wording, preserves the operator's answer). Targets carry a `lifecycle` (`registered → exploring → awaiting-answers → configured → testing → re-explore`) via `set_target_lifecycle`; `questionnaire_status` rolls up counts for the UI/lifecycle gate.
   - ✅ **Questionnaire API + UI built**: FastAPI endpoints under `/api/site/targets/{id}/questions` (list+rollup+lifecycle, create, `/answer`, `/skip`, delete) + `/lifecycle`; a **Questions tab** in the control room's Site Target view (grouped by category, per-kind answer inputs, secret answers masked as "•••• (vaulted)", Skip, and a lifecycle selector). Secret answers are vaulted server-side and never echoed.
   - ✅ **Capabilities P1+P2** (`docs/CAPABILITIES.md`): the "grant deeper access" layer. `qa_store/capabilities.py` — a seeded-but-extensible `capability_catalog` (30 baseline capabilities, 9 categories) + `site_capabilities` grants (secrets → vault, pointer only) + a derived **depth score** (access ladder, Black-box → Environment control). API: catalog/grants/depth/grant/decline/revoke + custom escape hatch. UI: a **Capabilities tab** (depth-ladder hero + header depth pill + per-card unlock/risk/Connect). After building, a **context-free fresh-eyes critique** flagged the screen as alarming (DB/kube access shown cold) — fixed in the same pass: a prominent "optional, you stay in control, only uses what you grant, vaulted" framing, sensitive L4–5 rungs collapsed behind an **"Advanced access"** expander, a depth HelpTip, and a lifecycle-badge legend. **Next (P4):** a granted capability lights up its MCP tool for the personas.
   - ✅ **Capabilities P3 + newcomer-trust fixes** (from a second context-free **fresh-eyes critique**, run as a general/non-infra user on a pristine stack): the explorer now **proposes the tailored shortlist** — `explore_target` maps detected affordances → `proposed_when` tags and sets matching capabilities `proposed` (earn-trust: never above L3, never clobbers an operator decision), surfaced as a **"✨ Suggested for this site"** section atop the Capabilities tab (excluded from the full ladder) with a "you can skip this entirely" framing. The critique also caught three trust-killers, all fixed: **New Run no longer defaults to the hardcoded `sandbox.slyreply.ai`** (a SlyReply cord-cut) — it defaults to a site you've added (`?target=<id>` deep-linkable) and Launch requires a target; **Settings** stopped claiming "nothing runs until this is set" (exploration needs no token) and now links **where to get the token**; and the explorer's homepage-title scrape stopped concatenating **inline-SVG `<title>` icon text** (it captures only the document title now). Proven live on a clean `docker compose up` + Playwright walkthrough (ghost.org → 7 tailored proposals, clean title, New Run targeting the user's site).
   - ✅ **Capabilities P4 — a granted capability lights up its MCP tool**: the bridge from the capabilities layer to the harness's persona runs, realising the `capabilities.py` docstring promise. `qa_agents/mcp_catalog.py` gains an `unlocked_by` field (which `capability_id`s light each server) + a `CAPABILITY_ENV` map (which env var a grant's vaulted credential populates) — wired today for **`email`** (← `sandbox-inbox`/`sandbox-sending`, `QA_MAILPIT_URL`) and **`openapi`** (← `openapi-spec`/`api-access`, `QA_OPENAPI_URL`). A new import-light resolver `qa_agents/target_mcp.py` (`resolve_target_mcp`, mirrors `site_knowledge.py`) reads a target's *granted* capabilities → the server ids to enable + their resolved vault creds. The run trigger (`POST /api/runs/trigger` now takes `target_id`) **augments** the run's enabled MCP set with those servers (never shrinking below the default-on set) and injects the creds as the env vars the harness **already reads** — so no `runner.py` change is needed. UI: each capability card shows **"🔌 Powers the … tool when granted"** and a granted+mapped one shows **"● active in runs"**; New Run shows **"N extra tools auto-enabled from this site's granted access"**; `GET /api/site/targets/{id}/mcp` serves the resolved server names (never secrets). Proven live: granting OpenAPI + sandbox-inbox on ghost.org through the UI lit both tools (depth → Instrumented inputs) and the target resolved `["email","openapi"]` for its runs. A **third fresh-eyes newcomer critique** confirmed the grant→tool→run chain lands (depth ladder + the matching-count New Run note are the wins) but flagged dev jargon and an ambiguous control — both fixed in the same pass: a `friendly_name` on each server ("email inbox tool" / "API probing tool") replaces "Mailpit" / "OpenAPI surface explorer" everywhere it's operator-facing, and the per-card "Not applicable" became a clearly-secondary **"Not for my site"** link (was a grey twin of the Connect button). Tests: store 405, api 339 (+5), harness 825 (+11), web 147 (+8). **Next:** map the remaining catalog rungs (logs/DB/admin) to their MCP servers as those integrations land; the LLM-driven explorer.
   - ✅ **Explorer v1 (heuristic)**: an **"Explore the site"** button + `POST /api/site/targets/{id}/explore` (`qa_review_api/explorer.py`). Fetches the registered site's homepage (GET-only — *discovery, not intrusion*), heuristically detects affordances (auth forms; signup/login/checkout/api links), writes **Surfaces / Flows / Knowledge**, **generates a tailored questionnaire**, and advances `registered → exploring → awaiting-answers`. No token needed (deterministic) — proven live against slyreply.ai (detected signup/login/checkout → 3 flows + 5 questions). **Next layer:** the LLM-powered, browser-driving explorer on the operator's Claude subscription (handles JS SPAs + richer discovery), then wiring the persona **run** from a `configured` site.
   - ✅ **BYOK credentials config (UI)**: a **Settings** page + `GET/PUT/DELETE /api/config/llm` to choose the LLM backend (`claude-code` default / `api`) and provide the credential it runs on. The token is a secret → **vaulted** (pointer only, never echoed); **environment vars are auto-detected** as a fallback source (the cluster pattern), and a UI-supplied token takes precedence. `qa_store.app_config` stores the choice + pointer; `get_llm_token` is the resolver the explorer/harness will read. This is the prerequisite for the explorer + persona runs (nothing runs without it).
   - ✅ **Onboarding/orientation UX redesign**: a real **"Add a site"** front door (`POST /api/site/targets`, slugified+deduped) + register modal on the **Sites** page (renamed from Site Model); nav reordered to the operator journey (Sites home, `/` → `/site`, Discovered promoted); a reusable **`HelpTip`** "?" popover explaining each concept in place (persona/site model/run/surfaces…); plain-language copy + guiding empty states; and an onboarding **lifecycle stepper** with a "Next: …" hint on the target page. **Next:** the explorer **agent** that probes a site and generates the model + questionnaire (registration creates the target at `registered`); then persona configuration + wiring an actual run.
3. **Testing personas on loops** — ~30 generic archetypes (now target-agnostic templates) read `test_flows` + the by-design `site_knowledge` from the Site Model and run.
4. **"Watch it drive"** — a live browser-streaming view of the agent driving Playwright, for trust/debugging. (Andrew: "would be amazing.")
5. **Semantic retrieval wiring** — use the vector layer (`site_retriever`) at targeted points (per-flow "what do we know about this area", triage dedup); run the reconciler in the embedding image (bake the fastembed model). Largely dormant until there's discovered content.
6. **GitHub-issue generation** — later; log internally first.

## Decisions that are settled — do NOT relitigate
- **BYOK + Claude Code flat-price is a MUST-HAVE.** Runs on the user's Claude Code Max OAuth token (flat price, not per-token); harness scrubs `ANTHROPIC_API_KEY` so the `claude` CLI uses the subscription; selectable `claude-code|api` backend; mint via `claude setup-token`. ToS gray-area caveat (subscription tokens for automated use; `api` is the sanctioned commercial path).
- **Embeddings default LOCAL/no-key** (`fastembed` `bge-small`, 384-d) because Anthropic has no embeddings API; OpenAI/Voyage optional via `qa-store[vector]`. atlas-local does local `$vectorSearch`.
- **Site knowledge is DATA, not code** — the Site Model (`site_targets`/`site_surfaces`/`test_flows`/`site_knowledge`), per-`(tenant,target)`. Flexible Mongo schema is deliberate.
- **Site-agnostic** — no SlyReply anywhere in the product.
